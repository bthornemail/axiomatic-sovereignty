## 1. Where The Pieces Come Together

This document is the living construction ledger for the foundation.

It names where reusable assemblies, forks, translations, and adaptations connect back to the five-layer structure. It does not turn those connections into a mandatory runtime, central product, or proprietary platform.

The five folders remain the pentagonal base:

```text
1-logic-proofs/         what is true
2-structural-types/     what is shaped
3-procedural-manifests/ how it moves
4-canonical-axioms/     what protects the sovereign entity
5-social-contracts/     what a community voluntarily agrees to
```

## 2. What A Construction Is

A construction is a named set of layer references that can be cloned, translated, implemented, audited, or forked.

Each construction should state:

- Name
- Purpose
- Layer references
- Required invariants
- Optional runtime interpretations
- Known forks or translations
- Consequences or corrections returned from use

Use relative paths. Do not reference local machine paths.

## 3. Construction Template

## 5. Branch Registry

Branches are not failures. They are adaptations that respond to specific needs, contexts, or visions.

Registration is visibility, not approval. A branch can exist without being listed here, and a listed branch does not become subordinate to this reference branch.

To list a branch:

1. Add an entry to this document, if you want the branch visible here.
2. State the branch name, purpose, and origin.
3. Link to the branch repository or distribution point.
4. State the branch relationship to the parent foundation.

### Branch Entry Template

```text
## Branch: [Name]

- Origin: [parent fork or foundation]
- Purpose: [why this branch exists]
- Language: [primary implementation language, if any]
- Status: [active, dormant, deprecated]
- Link: [repository URL]
- Delta: [what changed from the parent]
```

## 6. Fork Responsibility

To fork the foundation:

1. Fork the repository or create a new branch.
2. State your reason for forking in your branch, or in this document if you want the fork listed here.
3. Link your fork to the parent foundation where useful.
4. Preserve provenance.
5. Mark meaningful deltas.

If a fork modifies the foundation, the modification creates a responsibility boundary for the delta introduced. It does not create control over the original foundation.

## 7. Construction Invariant

```text
constructions are assemblies, not control points
assemblies use relative layer references
runtime interpretations are optional
branch registration is optional
forks may modify assemblies
modified assemblies create responsibility for their deltas
returned consequence improves the foundation
```
